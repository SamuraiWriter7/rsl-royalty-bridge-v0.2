# Migration Guide: v0.1 to v0.2

## Overview

This document explains how to migrate bridge implementations from
`RSL-to-Kazene Royalty Bridge Specification v0.1`
to
`RSL-to-Kazene Royalty Bridge Specification v0.2`.

The short version is:

- v0.1 defined the bridge concept
- v0.2 defines the bridge as an operationally reliable protocol

In other words:

- v0.1 = conceptual linkage
- v0.2 = auditable, replay-safe, failure-aware linkage

The purpose of this migration is not to change the philosophy of the bridge.
It is to make the bridge safe enough for real adoption.

---

## Why v0.2 Exists

v0.1 was already strong as a connection point between:

- external licensing and charging records on the RSL side
- internal trace and royalty allocation records on the Kazene side

However, v0.1 left several implementation-critical questions too loose:

- How should one-sided sync be handled?
- How should signatures and verification be expressed?
- How should duplicate replay be prevented?
- How should blocked records differ from rejected records?
- How should extensions evolve without breaking settlement logic?

v0.2 answers those questions.

Its role is not to widen the worldview.
Its role is to harden the bridge.

---

## Migration Goals

A successful migration to v0.2 should achieve the following:

1. introduce explicit bidirectional synchronization identifiers
2. introduce audit-oriented signature and verification structure
3. introduce idempotency and recomputation handling
4. introduce formal state transition semantics
5. introduce compatibility levels and extension namespaces
6. preserve v0.1 lineage without pretending that old records were already v0.2-safe

---

## High-Level Change Summary

## v0.1

v0.1 primarily expressed:

- conceptual linkage between RSL and Kazene layers
- access, trace, and settlement as related objects
- bridge semantics at a structural level

## v0.2

v0.2 adds formal operational controls:

- `sync_ids`
- `idempotency`
- `verification`
- `compatibility`
- formal `status` transition semantics
- stronger settlement gating rules

This means v0.2 is not merely “v0.1 plus more fields”.

It is a stricter contract.

---

## What Changes Structurally

## 1. Synchronization Becomes Explicit

### v0.1
Synchronization might be implied by contextual linkage.

### v0.2
Synchronization must be expressed explicitly through:

- `sync_ids.rsl_report_id`
- `sync_ids.trace_event_id`
- `sync_ids.inference_event_id`
- `sync_ids.external_charge_event_id`
- `sync_ids.royalty_settlement_id`
- `sync_ids.correlation_id`

### Migration meaning
If a v0.1 record only implied linkage informally, migration must either:

- backfill explicit sync identifiers, or
- mark the record as incomplete for settlement purposes

---

## 2. Verification Becomes a First-Class Object

### v0.1
Integrity assumptions may have existed outside the record.

### v0.2
Integrity state is part of the record itself:

- `verification.signature_bundle`
- `verification.verification_result`

### Migration meaning
Even if full cryptographic verification is not yet live, v0.2 requires
the structure to exist.

A migrated record may use placeholder verification state such as:

- `status = not_checked`
- signer metadata if known
- null verification timestamp if not yet executed

---

## 3. Replay Safety Becomes Explicit

### v0.1
Retry behavior may have depended on implementation convention.

### v0.2
Replay safety is formalized through:

- `idempotency.idempotency_key`
- `idempotency.dedupe_scope`
- `idempotency.dedupe_hash`
- `idempotency.replay_sequence`
- optional `idempotency.recomputation`

### Migration meaning
A v0.1 record cannot be treated as safely replay-aware unless these fields
are added or derived.

This is one of the main reasons why v0.1 records should not be auto-settled
without enrichment.

---

## 4. Failure States Become Formal

### v0.1
Incomplete, blocked, and invalid records may have been treated similarly.

### v0.2
These are now different states:

- `pending`
- `blocked_or_pending`
- `rejected`

### Migration meaning
Existing implementations that only track “open” and “error” should be
refactored into the richer state model.

Without this change, the bridge remains fragile in real operations.

---

## 5. Extension Handling Becomes Governed

### v0.1
Extensions may have been loosely added.

### v0.2
Extensions must declare:

- `extension_namespace`
- `compatibility_level`

And the record must declare:

- `compatibility.spec_family`
- `compatibility.compatibility_level`
- `compatibility.extension_namespace`

### Migration meaning
Custom fields can still exist, but they must stop pretending to be invisible.

v0.2 allows extensibility, but not extension chaos.

---

## Field-by-Field Migration Mapping

The following table shows the practical migration path.

| v0.1 concept | v0.2 destination | Migration action |
|---|---|---|
| conceptual bridge version | `bridge_version` | set to `0.2` only after transformation is complete |
| implicit linkage | `sync_ids.*` | backfill explicit identifiers |
| retry behavior by convention | `idempotency.*` | derive or generate idempotency metadata |
| integrity assumed externally | `verification.*` | create structured verification block |
| loose implementation status | `status` | map into formal v0.2 state model |
| extension-like ad hoc fields | `extensions[]` + `compatibility.*` | namespace and classify them |
| settlement outcome | `royalty_settlement.*` | preserve, but validate against v0.2 gating rules |

## Recommended Default Interpretation of Legacy v0.1 Records

When a v0.1 record is first read by a v0.2-compatible system, the safest default interpretation is:

- `bridge_version`: remain `0.1` until transformed
- interpreted compatibility: `compatible`
- interpreted verification status: `not_checked`
- interpreted operational state: `pending`

This preserves lineage without granting false confidence.

A v0.1 record should not magically become “ready” just because the new system can parse it.

---

## Recommended Migration Strategy

## Strategy A: In-Place Enrichment

Use this when:

- historical v0.1 records are already trusted enough to keep
- identifiers can be backfilled reliably
- the implementation needs continuity with minimal storage duplication

Approach:

1. read v0.1 record
2. generate missing v0.2 structures
3. keep provenance of migration
4. write enriched record as v0.2-compatible output

Recommended extra metadata:

- `migration.source_version`
- `migration.migrated_at`
- `migration.migration_tool`
- `migration.notes`

---

## Strategy B: Side-by-Side Re-Emission

Use this when:

- v0.1 historical shape should remain immutable
- audit requirements demand original preservation
- migration logic may evolve over time

Approach:

1. keep original v0.1 record unchanged
2. emit a new v0.2 record derived from it
3. link both through lineage metadata

Recommended extra metadata:

- `migration.source_record_id`
- `migration.source_version`
- `migration.transformation_id`

This is usually the strongest approach for auditable systems.

---

## Strategy C: Forward-Only Cutover

Use this when:

- old records do not need automated settlement
- only new ingestion flows need v0.2 guarantees
- migration cost for historical data is not justified

Approach:

1. freeze v0.1 as legacy
2. start generating only v0.2 for new flows
3. treat v0.1 data as reference-only unless manually enriched

This is the simplest operational path, though not the richest.

---

## Recommended Migration Phases

## Phase 1: Structural Compatibility

Goal:
Make v0.1 flows readable by v0.2-aware systems.

Tasks:

- add `sync_ids`
- add `compatibility`
- add `verification` placeholders
- add `idempotency` placeholders where derivable
- map status into `pending` by default

At this phase, the system becomes structurally bilingual.

---

## Phase 2: Operational Hardening

Goal:
Make the bridge safe under retries, sync gaps, and verification checks.

Tasks:

- define real `idempotency_key` generation
- define `dedupe_hash` normalization
- introduce timeout policy for `pending`
- introduce escalation policy to `blocked_or_pending`
- emit stable `reason_codes`

At this phase, the system stops being optimistic and starts being disciplined.

---

## Phase 3: Verification Activation

Goal:
Turn verification from structure into practice.

Tasks:

- start signing `access_snapshot`
- start signing `external_charge_event`
- start signing `royalty_settlement`
- start recording `verification_result`
- block settlement on terminal integrity failure

At this phase, the bridge becomes audit-capable.

---

## Phase 4: Automated Settlement Eligibility

Goal:
Allow safe automation for `ready` and `settled`.

Tasks:

- require sync pair presence
- require idempotency metadata
- require acceptable verification state
- prevent auto-settlement for `informational` and `experimental`
- support reversal and recomputation lineage

At this phase, v0.2 becomes a real operational protocol.

---

## Status Mapping Guidance

Legacy implementations often need a direct mapping.

### Recommended mapping

| Legacy condition | v0.2 status |
|---|---|
| record ingested, awaiting data | `pending` |
| record present, dependency missing beyond window | `blocked_or_pending` |
| record present, verification failed but review possible | `blocked_or_pending` |
| record definitively invalid | `rejected` |
| record fully synchronized and eligible | `ready` |
| settlement completed | `settled` |
| prior settlement corrected or undone | `reversed` |

### Important note
Do not map every historical “error” to `rejected`.

Many legacy “errors” were really unresolved incompleteness.
Those belong in `blocked_or_pending`, not final exclusion.

---

## Migration Rules for Historical Settlement Records

Historical records are the most dangerous part of migration because they create
the temptation to bless old uncertainty as new certainty.

Recommended rule set:

1. if historical record lacks both sync anchors, do not auto-upgrade to settlement-eligible
2. if historical record lacks idempotency metadata, do not treat replay history as safe
3. if historical record lacks verification structure, default to `not_checked`
4. if historical settlement lineage is unclear, keep it as legacy reference or require manual review
5. if historical record was already paid or settled externally, preserve that fact but distinguish it from v0.2-ready compliance

The bridge should preserve history, not mythologize it.

---

## Handling Missing Data During Migration

A migration process should define what to do when required v0.2 data cannot be recovered.

## Case 1: Missing `trace_event_id`
Recommended result:

- migrate record as `pending` or `blocked_or_pending`
- do not mark as `ready`

## Case 2: Missing `rsl_report_id`
Recommended result:

- migrate record as `pending` or `blocked_or_pending`
- escalate if the record depends on settlement

## Case 3: Missing both sync anchors
Recommended result:

- migrate record as `rejected`
- preserve audit lineage

## Case 4: Missing idempotency metadata
Recommended result:

- generate placeholder only if safe and documented
- otherwise require manual enrichment or keep out of auto-settlement

## Case 5: Missing verification inputs
Recommended result:

- emit `verification_result.status = not_checked`
- do not pretend the record is cryptographically trusted

---

## Extension Migration Guidance

Legacy implementations often contain custom fields that were never formally categorized.

During migration:

1. identify all non-core legacy fields
2. decide whether each field is:
   - settlement-critical
   - informational
   - experimental
3. assign `extension_namespace`
4. assign `compatibility_level`
5. move the field into `extensions[]` if it does not belong in the core

### Rule of caution
A field should not become settlement-critical merely because it already exists.

Criticality must be explicitly justified.

---

## Idempotency Migration Guidance

One of the hardest migrations is replay safety.

Recommended order:

1. define canonical business event units
2. define `dedupe_scope`
3. define deterministic normalization inputs
4. derive `dedupe_hash`
5. define stable `idempotency_key`
6. define recomputation policy for exceptions

### Safe default
If replay safety cannot be reconstructed confidently, do not grant automated settlement eligibility.

It is better to be slower than to pay twice with great confidence.

---

## Verification Migration Guidance

v0.2 does not require that every historical record instantly become cryptographically complete.

It does require honesty about verification state.

### Recommended legacy defaults

- `signature_bundle` may contain placeholders or known metadata
- `verification_result.status = not_checked`
- `checked_at = null`
- `verifier_id = null`
- `failure_reason = null`

### Important principle
Missing verification is not the same as failed verification.

Do not collapse them into the same meaning.

Validation After Migration

After migration, validate records in this order:

schema validity
does the record pass rsl-royalty-bridge-v0.2.schema.json?
state validity
does the declared status match the actual record contents?
sync validity
are rsl_report_id and trace_event_id aligned where required?
idempotency validity
are replay keys and hashes deterministic and non-empty?
verification validity
is the verification state honest and structurally complete?
compatibility validity
are extensions namespaced and correctly classified?
settlement validity
is the record truly eligible, or merely present?

A migrated record is not successful just because it parses.

It is successful when it behaves correctly.

Anti-Patterns to Avoid

Avoid these migration mistakes:

1. False readiness

Do not map parseable records directly to ready.

2. Retroactive certainty

Do not pretend historical records had sync, replay safety, or integrity guarantees they never had.

3. Extension smuggling

Do not allow unnamed custom fields to keep influencing settlement silently.

4. Error flattening

Do not collapse every legacy problem into a single “error” bucket.

5. Silent overwrite

Do not replace historical settlement lineage without preserving provenance.

6. Verification theater

Do not add signature-shaped fields without recording whether verification actually happened.

A bad migration makes the bridge look modern while remaining operationally medieval.

Recommended Minimal Migration Policy

For teams that want a clear baseline, the following policy is recommended:

parse v0.1 records as legacy inputs
preserve original lineage
emit v0.2 structure explicitly
default verification to not_checked
default unsettled migrated records to pending
use blocked_or_pending when missing data persists or replay safety is uncertain
reserve rejected for definitively invalid cases
do not auto-settle migrated legacy records without explicit enrichment

This policy is conservative, but safe.

That is the right bias for a bridge touching money, rights, and trace lineage.

Final Principle

v0.2 migration is not a cosmetic upgrade.

It is a shift from:

implied linkage to explicit synchronization
informal trust to structured verification
retry optimism to idempotent discipline
vague errors to formal state transitions
extension sprawl to governed compatibility

The migration succeeds when the bridge becomes harder to misuse,
not merely easier to describe.

That is the real meaning of moving from v0.1 to v0.2.
